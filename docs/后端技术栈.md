# 后端技术栈 (Backend Tech Stack)

本项目采用全栈单体仓库 (Monorepo) 模式，后端代码统一存放于 `backend/` 目录下。

后端的架构核心是构建一个**路由式 Agent 框架**，将网络入口、中枢大脑与具体的技能执行分层处理。

## 核心架构分层

### 1. 入口与网络层: FastAPI
- **职责**: 负责暴露 REST API 或 WebSocket 给前端调用，处理鉴权、请求基础校验，并管理底层数据库连接。
- **框架**: `FastAPI` + `uvicorn`
- **代码规范**:
  - 严禁在 `main.py` 堆砌逻辑，必须使用 `APIRouter` 按业务领域进行路由拆分。
  - 所有请求的 Request / Response 必须使用 Pydantic Model 强类型定义。

### 2. 中枢与大脑层: LangGraph
- **职责**: 作为 AI 系统的主路由节点 (Supervisor)。它不负责生成最终的内容，而是负责意图识别 (Intent Recognition) 和状态图 (State Graph) 的走向控制。
- **交互逻辑**:
  - 接收 FastAPI 传递过来的用户意图。
  - 维持图 (Graph) 中的上下文状态 (State)。
  - 根据用户的 Prompt，判定应该触发哪一个具体的下游 Agent (例如：规划 Agent、代码 Agent、闲聊 Agent)。
- **代码规范**:
  - LangGraph 的 `State` 定义必须轻量化。
  - `Node` (节点) 函数必须与业务逻辑解耦，保持路由清晰。

### 3. 业务执行层: Dify
- **职责**: 作为实际“干活”的平台。我们不在代码里硬编码复杂的提示词链路，而是把具体的 Agent 业务工作流托管给 Dify。
- **交互机制**:
  - LangGraph 在决定走向后，通过 HTTP 请求 (REST API) 远程调用 Dify 暴露的对应 Workflow/Agent 接口。
  - Dify 内部进行 RAG、大模型调用、工具调用。
  - FastAPI 层拿到 Dify 返回的数据（流式或块式），再封装返回给前端。

---

## 数据库与持久化

- **数据库**: `SQLite` (得益于极其轻量的特性，适合作为单文件持久化存储)。
- **ORM 框架**: **SQLModel**。
  - 它是 FastAPI 官方推荐的结合体，同时包含了 `Pydantic` 的接口校验能力与 `SQLAlchemy` 的底层数据库映射能力。
  - 极大减少代码样板，只需定义一次 Model 即可同时用于 API 验证与数据表创建。

## 目录结构参考 (后端部分)

```text
backend/
├── app/
│   ├── api/             # FastAPI 路由控制器
│   ├── core/            # 核心配置 (Config, Security)
│   ├── models/          # SQLModel 数据库模型
│   ├── services/        # 业务逻辑服务
│   ├── agents/          # LangGraph 图定义与节点逻辑
│   └── main.py          # FastAPI 应用入口
├── tests/               # 单元测试
├── pyproject.toml       # 依赖管理
└── requirements.txt     # (或锁文件)
```
