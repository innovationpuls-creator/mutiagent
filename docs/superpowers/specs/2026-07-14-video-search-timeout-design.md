# 视频检索超时收敛设计

## 目标

让章节视频检索在外部平台变慢或不可达时，在明确的总时限内返回可持久化的 `unavailable` 结果，不再让整条聊天流持续等待超过 Nginx 的 300 秒上游时限。

## 根因

视频检索对每个 brief 最多执行 6 个查询，每个查询并行访问 Bilibili 和 YouTube；每个小节最多进行 2 轮完整验证。当前只有单次 HTTP 请求的 8 秒超时，没有小节级或章节级总预算。7 个小节以 3 个并发执行，所有结果通过一次 `asyncio.gather` 汇总后才向上游返回，因此外部检索失败会累积成很长的等待。

## 方案

1. 保留现有的 Bilibili/YouTube 双平台查询和质量校验规则。
2. 为单个小节的视频检索增加明确的总预算；预算耗尽时记录超时原因并返回 `status: unavailable`。
3. 让每个小节独立收敛，单个小节失败不取消其他小节。
4. 保留现有的 3 小节并发限制，避免一次性压垮外部平台。
5. 为超时和外部请求失败增加结构化日志，包含 `section_id`、平台、查询词和耗时。
6. 用契约测试覆盖小节预算、失败收敛、并发结果汇总和原有成功路径。

## 不在范围内

- 不修改 Nginx 超时时间。
- 不改变视频质量判定规则。
- 不引入新的搜索供应商。
- 不修改前端事件协议。

## 验收标准

- 外部搜索持续等待时，单个小节在预算内返回 `unavailable`。
- 7 个小节的总等待时间由小节预算和并发限制决定，不再由 6 次查询乘以 2 轮重试无限累积。
- 现有视频成功、空结果和质量失败测试保持通过。
- 日志能区分外部搜索超时与质量校验失败。
